## מה זה API?

**API** (**Application Programming Interface**) הוא הדרך שבה תוכנות מדברות אחת עם השנייה. לא דרך מסך, לא דרך עכבר — דרך **הודעות טקסט מובנות**.

תחשבו על מסעדה. אתם לא נכנסים למטבח ומבשלים לבד. אתם יושבים בשולחן, מזמינים מהתפריט, והמלצר מביא את האוכל. ה-**API** הוא המלצר — הממשק בין מה שאתם רוצים לבין מה שהמטבח יודע לעשות.

בעולם התוכנה זה עובד אותו דבר: אפליקציה אחת שולחת **בקשה** (Request) לשרת, והשרת מחזיר **תשובה** (Response).

## Request & Response — הבקשה והתשובה

כל שיחה עם API מורכבת משני חלקים:

**Request (בקשה)** — מה שאתם שולחים. כולל:
- **Method** — מה אתם רוצים לעשות: `GET` (תן לי מידע), `POST` (קבל מידע חדש), `PUT` (עדכן), `DELETE` (מחק)
- **Endpoint** — הכתובת. למשל: `https://api.weather.com/tel-aviv`
- **Headers** — מטא-דאטה, כמו מפתח ה-API שלכם
- **Body** — המידע שאתם שולחים (בעיקר ב-POST)

**Response (תשובה)** — מה שחוזר. כולל:
- **Status Code** — מספר שאומר מה קרה: `200` = הכל בסדר, `404` = לא נמצא, `401` = לא מורשה, `500` = שגיאה בשרת
- **Body** — המידע עצמו, בדרך כלל בפורמט **JSON**

## JSON — השפה המשותפת

**JSON** (JavaScript Object Notation) הוא הפורמט שבו רוב ה-APIs מדברים. הוא נראה ככה:

```json
{
  "city": "Tel Aviv",
  "temperature": 28,
  "condition": "sunny"
}
```

זה פשוט טקסט מובנה — זוגות של מפתח וערך. קל לקרוא, קל לכתוב, וכל שפת תכנות יודעת לעבד אותו.

## Endpoint — הכתובת של השירות

ה-**Endpoint** הוא הכתובת הספציפית שאליה שולחים בקשה. כל שירות חושף כמה Endpoints:

```
GET  /users       → קבל רשימת משתמשים
GET  /users/42    → קבל את המשתמש עם ID 42
POST /messages    → שלח הודעה חדשה
```

תחשבו על Endpoint כמו **מספר טלפון**. כל מחלקה בארגון יש לה מספר משלה — אתם מתקשרים למספר הנכון לפי מה שאתם צריכים.

## API Key — תעודת הזהות

רוב ה-APIs דורשים **API Key** — מחרוזת ייחודית שמזהה אתכם. בלי מפתח, השרת לא יענה לכם.

למה? כי ה-API הוא משאב שמישהו משלם עליו. המפתח מאפשר לספק:
- לדעת **מי** שולח בקשות
- להגביל **כמה** בקשות מותר (Rate Limiting)
- לחסום שימוש לא מורשה

כשאתם נרשמים ל-OpenAI, Anthropic, או כל שירות API — אתם מקבלים מפתח. השמירה עליו היא **אחריות שלכם**. מפתח שנחשף = מישהו אחר יכול להשתמש בחשבון שלכם.

## איך הסוכן משתמש ב-API?

זוכרים את ה-[Function Calling](#chapter:agents-function-calling:function-calling-לתת-לסוכן-כלים) מפרק 5? שם למדנו שהסוכן "קורא לפונקציות". עכשיו אפשר לחבר את הנקודות: **הפונקציות האלה הן קריאות API**.

כשהסוכן מחליט לבדוק את מזג האוויר, הוא לא גולש לאתר. הוא שולח:

```
GET https://api.weather.com/tel-aviv
Authorization: Bearer sk-abc123...
```

ומקבל חזרה JSON עם הטמפרטורה. זה בדיוק מה שקורה כשהסוכן "משתמש בכלי" — מאחורי הקלעים, הוא שולח בקשת API ומעבד את התשובה.

## REST — הסטנדרט

רוב ה-APIs שתפגשו הם **REST APIs** (Representational State Transfer). REST הוא לא טכנולוגיה — הוא **סט כללים**:

1. כל משאב יש לו **כתובת** (URL)
2. משתמשים ב-**Methods** סטנדרטיים (GET, POST, PUT, DELETE)
3. כל בקשה היא **עצמאית** — השרת לא זוכר בקשות קודמות
4. התשובות בפורמט **אחיד** (בדרך כלל JSON)

אם ה-API של הסוכן שלכם עוקב אחרי הכללים האלה — כל מפתח בעולם ידע איך לעבוד איתו בלי לקרוא ספר הדרכה.

## למה זה חשוב לבונים סוכנים?

API הוא **הידיים של הסוכן**. בלי API, הסוכן יכול רק לדבר. עם API, הוא יכול:
- לשלוח מיילים
- לקרוא מבסיסי נתונים
- לבדוק מזג אוויר
- ליצור משימות ב-Jira
- לשלוח הודעות ב-Slack

בפרק הבא נראה איך לחבר את זה בפרקטיקה — עם **n8n** ו-**Docker**, שהופכים קריאות API לתהליכים אוטומטיים שלמים.
